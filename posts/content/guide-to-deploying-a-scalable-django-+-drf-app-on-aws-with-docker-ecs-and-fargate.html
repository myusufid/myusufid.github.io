Guide to Deploying a Scalable Django + DRF App on AWS with Docker, ECS, and Fargate

2026-02-18

Deploying a Django + DRF app in production isn’t as straightforward as running python manage.py runserver. For local development, Django's built-in development server works well enough, but it's not designed to handle high traffic, scale across multiple servers, or integrate with complex cloud infrastructure.

To deploy a production-ready Django app that is scalable, secure, and highly available, we need a more robust infrastructure. This is where Docker, AWS ECS, Fargate, and Kubernetes come in.

In this guide, we will create a containerized Django application, push it to a container registry (AWS ECR), deploy it on AWS using ECS with Fargate, and set up automatic scaling, load balancing, and monitoring. We'll also secure the app with HTTPS using AWS ACM (Certificate Manager), and store static/media files in AWS S3. Finally, we’ll explore how Kubernetes fits into the picture and why you may want to use it over ECS for complex setups.

What You Will Learn:

  • Dockerize a Django app
  • Create an AWS ECR repository
  • Deploy the app on AWS ECS using Fargate
  • Configure load balancing using AWS ALB
  • Set up auto-scaling
  • Secure the app with HTTPS using AWS ACM
  • Optionally deploy using Kubernetes

1. Set Up a Django + DRF Project

Let’s begin by creating a new Django project and setting up DRF.

Step 1.1: Create a New Django Project

Create a project directory and a virtual environment:

mkdir django-aws-deploy
cd django-aws-deploy
python -m venv env
source env/bin/activate

Step 1.2: Install Django and DRF

Install Django, DRF, and Gunicorn:

pip install django djangorestframework gunicorn psycopg2-binary

Create a new Django project and a DRF app:

django-admin startproject myproject .
python manage.py startapp api

Step 1.3: Configure DRF

Add rest_framework to INSTALLED_APPS in settings.py:

# myproject/settings.py

INSTALLED_APPS = [
    # ...
    'rest_framework',
    'api',
]

Step 1.4: Create a Sample DRF Endpoint

Create a simple hello world endpoint to test our deployment later.

api/views.py

from rest_framework.decorators import api_view
from rest_framework.response import Response

@api_view(['GET'])
def hello_world(request):
    return Response({'message': 'Hello World!'})

api/urls.py

from django.urls import path
from .views import hello_world

urlpatterns = [
    path('hello/', hello_world),
]

myproject/urls.py

from django.contrib import admin
from django.urls import path, include

urlpatterns = [
    path('admin/', admin.site.urls),
    path('api/', include('api.urls')),
]

Test the endpoint locally:

python manage.py runserver

Navigate to http://127.0.0.1:8000/api/hello/ to confirm it works.

2. Dockerize the Django Project

To ensure our app runs consistently across different environments, we need to containerize it using Docker.

Step 2.1: Create a Dockerfile

Create a Dockerfile in the root of the project:

# Use official Python image
FROM python:3.11-slim

# Environment settings
ENV PYTHONDONTWRITEBYTECODE 1
ENV PYTHONUNBUFFERED 1

# Set working directory
WORKDIR /app

# Install dependencies
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copy project files
COPY . .

# Collect static files
RUN python manage.py collectstatic --noinput

# Expose port
EXPOSE 8000

# Start server with Gunicorn
CMD ["gunicorn", "myproject.wsgi", "--bind", "0.0.0.0:8000", "--workers", "4"]

Optimizing Gunicorn Configuration

When deploying to production, simple defaults might not be enough. For example, if you are using a 2 vCPU / 4 GiB instance on ECS, you should calculate the workers based on the CPU cores.

The standard formula is (2 × CPU cores) + 1. For 2 vCPUs, this gives you 5 workers.

A configuration of 3 workers might be underutilized on such hardware. 5 is the sweet spot — it keeps all cores busy without over-provisioning memory. Each Gunicorn worker for a Django app typically uses 50–150 MB, so 5 workers would consume roughly 250–750 MB, well within a 4 GiB limit.

CMD ["gunicorn", "--bind", "0.0.0.0:8000", "--workers", "5", "--threads", "2", "--timeout", "120", "myproject.wsgi:application"]

Adding --timeout 120 and --threads 2 helps if your app handles any slow requests or I/O-heavy views. This effectively gives you 10 concurrent execution slots (5 workers × 2 threads) without spawning more heavyweight processes.

Step 2.2: Create Docker Compose for Local Development

Create a docker-compose.yml file to orchestrate the services locally.

version: '3.8'

services:

volumes:
  postgres_data:

Exclude unnecessary files from Docker builds:

__pycache__

Step 2.4: Build and Run

docker-compose build
docker-compose up

Step 2.6: Tag and Push Docker Image

Assuming you have created an ECR repository, authenticate and push your image:

# Authenticate
aws ecr get-login-password --region us-east-1 | docker login --username AWS --password-stdin .dkr.ecr.us-east-1.amazonaws.com

# Tag
docker tag myproject:latest .dkr.ecr.us-east-1.amazonaws.com/myproject:latest

# Push
docker push .dkr.ecr.us-east-1.amazonaws.com/myproject:latest

Why AWS ECS + Fargate?

Elastic Container Service (ECS) is Amazon’s native container orchestration service. It is tightly integrated with AWS infrastructure. Fargate allows you to run ECS containers without managing EC2 instances.

  • No need to provision EC2 instances
  • Fully managed by AWS
  • Scales automatically based on demand
  • Secure by default using IAM and VPC

3. Create an ECS Cluster and Task Definition

Create an ECS Cluster

  1. Go to Elastic Container Service (ECS) → Clusters → Create Cluster.
  2. Select "Networking Only" (Fargate).
  3. Name the cluster django-cluster and create it.

Create a Task Definition

A task definition is a blueprint that defines how a container should run.

  1. Go to ECS → Task Definitions → Create Task Definition.
  2. Choose Fargate.
  3. Set Task Name to django-task.
  4. Task Role: Create a new role (ecsTaskExecutionRole).
  5. Network Mode: awsvpc.
  6. CPU: 512 (.5 vCPU), Memory: 1024 (1 GB).

Define Container Settings

  • Container Name: django-app
  • Image: <account-id>.dkr.ecr.us-east-1.amazonaws.com/myproject:latest
  • Port Mappings: 8000

Environment Variables

Pass necessary environment variables such as DEBUG, ALLOWED_HOSTS, DATABASE_URL, and SECRET_KEY (preferably retrieved from Secrets Manager).

4. Create an ECS Service

An ECS Service allows you to run and maintain a specified number of instances of a task definition.

  1. Go to ECS → Create Service.
  2. Launch Type: Fargate.
  3. Cluster: django-cluster.
  4. Number of tasks: 2 (for high availability).
  5. Select an existing VPC and at least two subnets in different Availability Zones.
  6. Create a new Security Group allowing inbound traffic on ports 80, 443, and 8000.

Enable Auto Scaling

Set up a policy to scale based on CPU utilization:

  • Scale up at 70% CPU.
  • Scale down at 30% CPU.
  • Min tasks: 2, Max tasks: 10.

5. Set Up an Application Load Balancer (ALB)

An ALB will distribute incoming traffic to the ECS tasks.

Create an ALB

  1. Go to EC2 → Load Balancers → Create Load Balancer.
  2. Choose Application Load Balancer.
  3. Scheme: Internet-facing.
  4. Select the same subnets used for the ECS service.

Create a Target Group and Listener

  1. Create a target group for HTTP traffic on port 8000 with protocol HTTP.
  2. Set the health check path to /api/health/ (we'll add this endpoint next).
  3. Add a listener to the ALB on port 80 forwarding to this target group.

6. Secure with HTTPS and Finalize

To secure your application, use AWS ACM to request a free SSL certificate. Add an HTTPS listener (port 443) to your ALB and forward traffic to the target group.

Update Django Settings

Ensure your Django settings enforce security:

SECURE_SSL_REDIRECT = True
SESSION_COOKIE_SECURE = True
CSRF_COOKIE_SECURE = True
ALLOWED_HOSTS = ['your-load-balancer-dns-name']

Add Health Check Endpoint

# api/views.py
@api_view(['GET'])
def health(request):
    return Response({"status": "healthy"})

# api/urls.py
urlpatterns = [
    # ...
    path('health/', health),
]

After pushing changes and updating the ECS service, your application will be live, scalable, and secure!

Kubernetes (Optional)

While ECS is excellent for many use cases, Kubernetes (EKS) offers even more control for complex, multi-container workloads. You can set up EKS using eksctl and deploy using standard Kubernetes manifests if your requirements outgrow ECS.
